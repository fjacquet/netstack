The form no longer uses a `FormValues` bridge type -- it works directly with `SizingInput` fields (except for the rack editor which manages the racks array as local state synced to the store).

**Section 1: Rack Configuration (RACK-01, RACK-02)**

Replace the `totalServers` and `rackCount` fields with:

1. A "Number of Racks" number input (min=1, max=200). When user changes this value:
   - If increased: append new rack entries with `serverCount: 16` (default)
   - If decreased: truncate from the end (remove last racks)
   - Call `setInput({ racks: updatedRacks })` immediately (debounce 150ms for number inputs)

2. Below the rack count, render a scrollable list of per-rack server count inputs:
   - Use a `div` with `max-h-48 overflow-y-auto space-y-2` for the rack list when rack count > 4
   - Each rack row: label "Rack N" (using `t('sizing.rackLabel', { n: i+1 })`) + number input (min=0, max=500)
   - When any per-rack server count changes (debounced 150ms): rebuild the full racks array and call `setInput({ racks: [...] })`
   - Show total server count as read-only summary text below the list: `t('sizing.totalServersDisplay', { count: totalServers })`

Remove the `totalServers` field entirely. Remove the `toRacksArray()` helper function. Remove the `FormValues` bridge interface.

**Form state management approach:**

- Use `useForm` with a `FormValues` type that has: `rackCount: number`, `rackServers: number[]` (parallel array of server counts), plus all existing scalar fields (`portsPerServerFrontend`, `portsPerServerBackend`, `activeUplinksPerLeaf`, `connectivityType`, `cableType`, `leafModel`, `spineModel`, `borderLeafModel`, `borderLeafCount`, `rackSize`).
- Initialize `rackCount` from `input.racks.length` and `rackServers` from `input.racks.map(r => r.serverCount)`.
- The `form.watch` effect syncs changes back to the store.
- When `rackCount` changes: resize the `rackServers` array (pad with 16, or truncate), then update store.
- When any `rackServers[i]` changes: rebuild racks array from `rackServers`, then update store.

**Section 2: Server Port Configuration (PORT-01, PORT-02)**

Add two number inputs after the rack section, grouped in their own visual section:

1. "Frontend Ports per Server" (`portsPerServerFrontend`): `type="number"`, min=0, max=8, default 1. When changed (debounced 150ms): `setInput({ portsPerServerFrontend: value })`.

2. "Backend Ports per Server" (`portsPerServerBackend`): `type="number"`, min=0, max=8, default 1. When changed (debounced 150ms): `setInput({ portsPerServerBackend: value })`.

Add a helper text below each field using FormDescription: `t('sizing.frontendPortsHelp')` and `t('sizing.backendPortsHelp')`.

**Section 3: Uplink Configuration (UPLN-01)**

Add after the Leaf Switch Model select:

1. "Active Uplinks per Leaf" number input (`activeUplinksPerLeaf`): `type="number"`, min=1, max dynamically set from `SWITCH_CATALOG[currentLeafModel].uplinkPorts`.
   - Import `SWITCH_CATALOG` from `@/domain/catalog/hardware`
   - Read `form.watch('leafModel')` to get the current leaf model
   - Derive `maxUplinks = SWITCH_CATALOG[currentLeafModel].uplinkPorts`
   - When leaf model changes and current `activeUplinksPerLeaf > maxUplinks`, clamp it down: `form.setValue('activeUplinksPerLeaf', maxUplinks)` and `setInput({ activeUplinksPerLeaf: maxUplinks })`
   - Show helper text: `t('sizing.activeUplinksHelp', { max: maxUplinks })`

**Form layout organization:**

Group fields into visual sections using semantic headings (h3 or bold FormLabel):

1. "Rack Configuration" section: rack count + per-rack server inputs + total display
2. "Server Connectivity" section: frontend ports, backend ports
3. "Network Configuration" section: connectivity type, cable type, leaf model, active uplinks, spine model
4. "Border Leaf" section: border leaf model, border leaf count (conditional)
5. "Physical" section: rack size
