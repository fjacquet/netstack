## Spacing Scale

Declared values (multiples of 4 only). No changes from previous phases.

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, inline padding, U-label padding |
| sm | 8px | Inter-slot gap in rack frame, badge padding |
| md | 16px | Section padding, card content padding |
| lg | 24px | Rack frame vertical padding (`py-6`) |
| xl | 32px | Layout gaps between rack frame and controls |
| 2xl | 48px | Major section breaks |
| 3xl | 64px | Page-level spacing |

Exceptions:

- Rack U-slot row height: 44px (h-11) — existing pattern from RackFrame.tsx, matches touch target minimum. Keep as-is.
- Rack device inner content: 40px (h-10) — 4px clearance from slot row boundary. Keep as-is.
- 1U server slot height: 44px — matches existing h-11 U-slot row, consistent with switch rendering.
- 2U server slot height: 88px — spans 2 U-slot rows (2 × 44px).
- 4U server slot height: 176px — spans 4 U-slot rows (4 × 44px).
- 8U server slot height: 352px — spans 8 U-slot rows (8 × 44px).

Source: `RackFrame.tsx` line 112 (`h-11`), confirmed as existing pattern.

## Interaction Contracts

### ELEV-02: U-height input field

**Location**: `InputForm.tsx` — inside the "Physical" section (`physicalHeading`), below `rackSize`.

**Control**: shadcn `Select` with 4 options (consistent with existing `rackSize` Select pattern).

```
Label:    "Server U-Height"       (i18n: sizing.serverUHeight)
Options:  1U / 2U / 4U / 8U
Default:  1U
Help text: none (self-explanatory)
```

**Rationale**: InputForm is the established input surface. Adding a Select here is consistent with `rackSize` and `leafModel`. Placing it in the rack elevation view would create a split input surface (violates single-source principle established in Phase 2).

**Schema field to add**: `serverUHeight: z.enum(['1U','2U','4U','8U']).default('1U')` in `SizingInputSchema`.

### ELEV-01: Server slot rendering

Servers are rendered inside `buildRackDevices()` above the switch devices (U4 and up):

- Default placement: U4 = first server group start (U1=OOB, U2=Leaf B, U3=Leaf A).
- One `ServerDevice` per server, stacked upward from U4.
- Each `ServerDevice` occupies `uHeight` consecutive U-slots. The rack frame must skip those slot numbers.
- Label: "Server N" where N is the 1-based index within the rack (Server 1, Server 2, …).
- Model text: omitted for servers (no model field in server config).

**RackFrame multi-slot device rendering**: The existing slot-by-slot rendering loop must be extended to support multi-U devices. When a device occupies `uSlot` through `uSlot + uHeight - 1`, render a single `ServerDevice` element with `height = uHeight * 44px` spanning those rows. Skip the covered slots in the loop.

### ELEV-03: RACK_CAPACITY_EXCEEDED alert

**Violation rendering location**: `BOMPanel.tsx` — Section D (Violations), matching the `OOB_PORT_SATURATION` and `SPINE_CAPACITY_EXCEEDED` pattern exactly.

**Alert variant**: `destructive` — uses `<AlertCircle>` icon (matching `OOB_PORT_SATURATION`).

**Rack frame visual indicator**: When the currently-selected rack's total device U-height exceeds `rackSize`, the rack frame outer border switches from `border-border` to `border-destructive`. This is a read-only visual affordance, not interactive.

**U-capacity overflow strip** (inside rack frame): A 4px-tall red stripe rendered below the last U-slot label, with text: "Overflow: +Xu" (where X = exceeded units). Only visible when capacity is exceeded.

**Per-rack violation scoping**: The engine emits one `RACK_CAPACITY_EXCEEDED` violation per rack that overflows. The BOM panel shows all violations. The rack frame visual indicator applies only to the currently-selected rack.

### Drag-and-drop constraint

Server device slots (`ServerDevice`) are NOT draggable in this phase. Only switch devices (leaf, spine, OOB, border) retain drag-and-drop reordering. Add a comment in `RackFrame.tsx` noting this is v2 scope.

## Design Questions — Explicit Answers

### Q1: Server slot appearance

- Background: `hsl(25 95% 80%)` light / `hsl(25 95% 28%)` dark (amber/orange, distinct from leaf=blue, spine=purple, oob=neutral, border=green).
- Border: uses `getSaturationBorderClass()` pattern with `usedPorts=0 / totalPorts=0` (returns no saturation border — clean border-2 border-border).
- Label: "Server N" (1-based index), shown as `text-xs text-foreground/80 truncate`.
- Model text: omitted (servers have no model).
- Icon: none. The amber color is sufficient distinguisher.
- U-height badge: small `text-[10px]` badge in top-right corner of the slot showing "Nu" (e.g. "2U").

### Q2: U-height rendering

- 1U: height 44px (`h-11`) — one slot row, same as current switch devices.
- 2U: height 88px (`h-[88px]`) — two slot rows.
- 4U: height 176px (`h-[176px]`) — four slot rows.
- 8U: height 352px (`h-[352px]`) — eight slot rows.
- The device content (label + badge) is centered vertically within the full height.
- Proportional: yes, each U = 44px exactly (matches existing h-11 slot row).

### Q3: U-height input field placement

- Location: `InputForm.tsx`, "Physical" section, below `rackSize` Select.
- Control: shadcn `Select` with 4 options (1U/2U/4U/8U).
- NOT in the rack elevation view — see rationale under ELEV-02 interaction contract above.

### Q4: RACK_CAPACITY_EXCEEDED alert placement

- Primary: `BOMPanel.tsx` Section D (Violations), using `Alert variant="destructive"` — identical pattern to `OOB_PORT_SATURATION`.
- Secondary visual: rack frame outer border turns destructive-red for the affected rack.
- STATE.md Research Flag confirmed: "RACK_CAPACITY_EXCEEDED violation is best surfaced in BOM panel vs inline" — decision confirmed, aligns with existing violation pattern.

### Q5: Overflow behavior

When total device U-height (all servers + 3 switches) exceeds `rackSize`:

1. Rack frame outer border: `border-destructive` (4px red border, replacing neutral `border-border`).
2. Overflow strip: 4px tall red bar at the bottom of the slot list, labeled "+Xu overflow".
3. BOM panel: destructive Alert with title + contextual body copy (rack number, used U, total U).
4. Individual slots: no change — slots do not turn red. The frame-level indicator is sufficient.
5. Rack capacity badge in selector bar: turns red, text appends "— overflow".
