| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps (`gap-1`), inline chevron padding |

- Touch target minimum: 44px for accordion triggers (combine `min-h-11` with padding).
- Accordion trigger padding: `py-4 px-6` (16px vertical, 24px horizontal) — consistent with Card padding conventions.

## Route Architecture

### Router Setup

Use `HashRouter` (not `BrowserRouter`) for GitHub Pages compatibility. HashRouter eliminates the need for a 404.html fallback file and is the simpler choice for a static deployment.

**Decision rationale (Claude's discretion):** ADR-0021 left HashRouter vs BrowserRouter open. HashRouter is chosen because: (1) GitHub Pages serves from a subdirectory path, (2) no server-side config available, (3) 404.html SPA redirect is fragile. HashRouter URLs (`/#/input`, `/#/topology`) are acceptable for a tooling app.

```tsx
// src/main.tsx — wrap at root level
import { HashRouter } from 'react-router-dom'

<HashRouter>
  <App />
</HashRouter>
```

Note: `basename` prop is NOT needed with HashRouter — hash routing is always relative to the root.

### Route Table

| Path | Component | Description |
|------|-----------|-------------|
| `/input` | `InputPage` | Dedicated accordion input form — new in Phase 24 |
| `/` | `ResultsPage` | BOM panel only — split from SizingPage |
| `/topology` | topology content | Unchanged — just moved from TabsContent |
| `/rack` | rack content | Unchanged — just moved from TabsContent |

### App.tsx Migration

Replace `<Tabs>` / `<TabsList>` / `<TabsContent>` structure with `<Routes>` / `<Route>` components. The `mode` and `profilesOpen` state remain in `AppContent`.

```tsx
// Conceptual structure after migration
<Routes>
  <Route path="/input" element={<InputPage mode={mode} />} />
  <Route path="/" element={<ResultsPage mode={mode} />} />
  <Route path="/topology" element={<TopologyContent mode={mode} />} />
  <Route path="/rack" element={<RackContent mode={mode} />} />
</Routes>
```

## Accordion Component

### Installation

```bash
npx shadcn add accordion
```

This creates `src/components/ui/accordion.tsx` using Radix UI `@radix-ui/react-accordion`. The standard shadcn/ui accordion implementation provides:
- `Accordion` — root with `type="single"` and `collapsible` props
- `AccordionItem` — each section wrapper with border-bottom
- `AccordionTrigger` — clickable header with animated ChevronDown
- `AccordionContent` — animated expand/collapse panel

Registry: shadcn official only. No third-party registries. Safety gate: not required.

### Accordion Configuration

```tsx
<Accordion type="single" collapsible defaultValue="rack-config" className="w-full">
  <AccordionItem value="rack-config">
    <AccordionTrigger className="text-base font-semibold px-6 py-4">
      {t('input.section.rackConfig')}
    </AccordionTrigger>
    <AccordionContent className="px-6 pb-6">
      {/* fields */}
    </AccordionContent>
  </AccordionItem>
  {/* additional items */}
</Accordion>
```

- `type="single"` — only one section open at a time (simpler UX for linear config flow)
- `collapsible` — allows closing the open section (user can collapse rack config after filling it)
- `defaultValue="rack-config"` — Rack Config section open on every initial load
- No localStorage persistence — open/closed state resets on page navigation

### Accordion Animation

Use shadcn/ui default: `data-[state=open]:animate-accordion-down data-[state=closed]:animate-accordion-up`. Duration: 200ms ease-out (Tailwind animation default). Do not customize animation timing in this phase.

### Accordion Trigger Design

```
[ Section Title                              ChevronDown ]
```

- Left: section title text (`text-base font-semibold`)
- Right: `ChevronDown` icon from lucide-react, rotates 180deg when open (standard shadcn behavior via `[data-state=open]>svg { transform: rotate(180deg) }`)
- Full-width click target (block-level trigger)
- No icon before section title — keep trigger clean. Icon could be added in a future phase.

## Input Page Layout (`/input` route)

### Page Structure

```
[TopBar — unchanged, h-11]
[Tab strip — NavLinks: Sizing | Topology | Rack — h-11]
[main — flex-1 overflow-auto]
  [div — max-w-3xl mx-auto px-6 py-8]
    [Page Title — text-xl font-semibold mb-1]
    [Page Description — text-sm text-muted-foreground mb-6]
    [Card — rounded-lg border bg-card]
      [Accordion — w-full]
        [AccordionItem × N]
```

**Primary visual anchor:** The open Rack Config accordion section immediately below the page title draws the eye to the first required input on load.

**Max width:** `max-w-3xl` (768px) — wide enough for comfortable two-column form layout at desktop, fits a single column below 768px. Centers on wide screens. Do not go full-width; this is a form, not a dashboard.

**Responsive behavior:**
- Below 768px: single-column field layout inside accordion sections
- 768px and above: two-column grid (`grid grid-cols-2 gap-4`) for Select/Input pairs within each section
- The Rack Config per-rack server inputs remain single-column at all widths (they are a repeating list)

**Page scroll:** The `main` element is `overflow-auto`. The accordion lives inside scrollable content — no sticky accordion header. Users scroll the page normally.

### Mode-Aware Content

The `InputPage` component receives `mode: 'ethernet' | 'fc' | 'converged'` as a prop (passed from `AppContent`). It renders the appropriate accordion form component:

```tsx
// InputPage.tsx
function InputPage({ mode }: { mode: 'ethernet' | 'fc' | 'converged' }) {
  return (
    <div className="max-w-3xl mx-auto px-6 py-8">
      <h1 className="text-xl font-semibold mb-1">{t('input.pageTitle')}</h1>
      <p className="text-sm text-muted-foreground mb-6">{t('input.pageDescription')}</p>
      <Card>
        {mode === 'ethernet' && <EthInputAccordion />}
        {mode === 'fc' && <FCInputAccordion />}
        {mode === 'converged' && <ConvergedInputAccordion />}
      </Card>
    </div>
  )
}
```

The topology selector (Clos vs Three-Tier) within Ethernet mode is rendered inside `EthInputAccordion` — it does not affect the outer page structure.

## Results Page (`/` route)

### Page Structure

`ResultsPage` is `SizingPage.tsx` with the `InputForm` removed. The BOM panel expands to full available width.

```tsx
// ResultsPage.tsx — replaces SizingPage.tsx
function ResultsPage({ mode }: { mode: 'ethernet' | 'fc' | 'converged' }) {
  return (
    <div className="p-6">
      {mode === 'fc' ? <FCBOMPanel /> : mode === 'converged' ? <ConvergedBOMPanel /> : <BOMPanel />}
    </div>
  )
}
```

**BOMPanel width:** `max-w-5xl mx-auto` — wider than the input page (1024px) because BOM tables need more horizontal space. Do not constrain to the same 768px as the input form.

**Empty state update:** The existing `results.emptyBody` copy says "Fill in the form on the left". This must be updated to reflect the new navigation. New copy: `"Configure your network in the Inputs page, then return here to see your BOM."` — add as `results.emptyBodyV2` i18n key (or update existing key). See Copywriting Contract below.

**Live recalculation:** No changes to Zustand subscriptions. The `resultStore` subscription to `inputStore` is in the store layer — it fires regardless of which route is active. Results update in the background while the user is on `/input`.


### Form Integration Pattern

Each accordion component hosts its own `useForm` instance scoped to the fields in that mode. The pattern matches `InputForm.tsx` exactly:

1. `useForm<FormValues>({ defaultValues: { ...from store } })`
2. `useEffect` to watch form and call `setInput` on every valid change
3. `useShallow` selector for store reads
4. All field components use `<FormField control={form.control} name="..." render={...} />`

Do not use a single global `useForm` across all 3 accordion components — keep them independent.
